A single call and its lifecycle.

public interface ICall

Remarks

Event threading contract. Event handlers run synchronously on the SDK thread that raised the event — this is usually not the thread that created the call. Handlers therefore must not block or perform long-running/synchronous I/O: a blocked handler stalls the SDK path that raised it (SIP signaling or media), delaying every other call on the same line. Off-load real work to your own task or queue, e.g. call.StateChanged += (_, e) => _ = Task.Run(() => Handle(e));. Handlers should also not throw. See each event for the specific thread and whether it is serialized.

TransferRequested is the exception to off-loading: it needs a synchronous accept/reject decision, so make that decision quickly inline (do not await long work).

Error contract. The action methods split into two error styles by return type. The core lifecycle operations return Task (or Task<TResult> of bool for AttendedTransferAsync(ICall, CancellationToken)) and throw: invalid usage surfaces as InvalidOperationException (wrong state or direction), ArgumentException (invalid argument), and cancellation surfaces as OperationCanceledException; transport, timeout and unexpected transport-layer failures propagate as-is. The extended in-dialog / inbound-response operations return CallActionResult instead and do not throw for foreseeable outcomes: an invalid state or direction, a protocol rejection (for example a SIP 4xx from the peer), an invalid request, cancellation, and transport failures are all reported as a non-success CallActionResult whose Status classifies the cause (see CallActionStatus). These result-returning methods are the ones that involve a SIP round-trip whose negative answer is a normal protocol result rather than a programming error. Per-method tags below state the exact exceptions and result semantics.

Events

DtmfReceived

Raised when a DTMF tone is received. This event may fire from two threads: the SIP signaling thread (SIP INFO tones) and the media receive thread (RFC 4733 in-band RTP events). It is therefore not guaranteed single-threaded — keep the handler thread-safe and fast.

event EventHandler<DtmfReceivedEventArgs>? DtmfReceived

Event Type

EventHandler<DtmfReceivedEventArgs>

HoldStateChanged

Raised when the remote party puts the call on or off hold. Serialized on the SIP signaling thread; buffered and replayed like StateChanged when no handler is attached.

event EventHandler<HoldStateChangedEventArgs>? HoldStateChanged

Event Type

EventHandler<HoldStateChangedEventArgs>

QualitySnapshotChanged

Raised periodically with an updated call-quality snapshot. Fires on a media/RTCP thread (RTCP send-timer and receive driven, not the signaling thread), so it may interleave with the signaling-thread events above. Not buffered.

event EventHandler<CallQualitySnapshotChangedEventArgs>? QualitySnapshotChanged

Event Type

EventHandler<CallQualitySnapshotChangedEventArgs>

StateChanged

Raised when State changes. Serialized on the SIP signaling thread. If no handler is attached yet, state changes are buffered and replayed in order once one is — the initial state is never missed. See the interface remarks for the handler contract.

event EventHandler<CallStateChangedEventArgs>? StateChanged

Event Type

EventHandler<CallStateChangedEventArgs>

TransferRequested

Raised when the peer requests a transfer (SIP REFER), on the SIP signaling thread. The handler must set the accept/reject decision on the event args synchronously (see remarks); it drives the SIP response, so decide quickly and do not await long-running work.

event EventHandler<TransferRequestedEventArgs>? TransferRequested
